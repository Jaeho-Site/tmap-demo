# 산책온 (SanchaekOn) — UX 원칙

> **문서 상태:** 초안 v1.0 · **작성일:** 2026-07-09
> **함께 볼 문서:** [PRD.md](./PRD.md), [USER_STORIES.md](./USER_STORIES.md)
> **목적:** 디자인의 북극성. 모든 화면, 카피 문구, 인터랙션은 이 원칙 중 하나로 추적 가능해야 한다. 두 원칙이 충돌할 때는 §1의 순서가 우선순위를 가른다.

---

## 1. 열 가지 원칙 (우선순위 순)

1. **수 초 안에 걷기 시작.** "걷고 싶다"와 "선택한 코스를 걷고 있다" 사이의 거리는 물리적으로 가능한 한 짧아야 한다.
2. **한 번에 하나의 결정.** 선택 마비를 막아라 — 옵션의 벽이 아니라, 강력한 추천 하나를 보여줘라.
3. **항상 이유를 설명하라.** 모든 추천은 실제 코스 데이터에 근거한 쉬운 말의 이유를 담는다.
4. **피드백은 탭이지 에세이가 아니다.** 제품 전체를 마찰 사다리의 가장 낮은 단에 맞춰 최적화하라.
5. **적절한 순간에 물어라.** 타이밍이 설득을 이긴다 — 기억이 가장 생생할 때 피드백을 요청하라.
6. **기여가 기분 좋게 하라 — 나에게도, 남에게도.** 상호성(내 다음 산책이 나아진다) + 이타심(내가 남을 도왔다).
7. **신뢰도에 정직하라.** 얇은 데이터로 과잉 약속하지 말고, 불확실성을 숨기지 말고 보여줘라.
8. **중독이 아니라 누적을 보상하라.** 리텐션은 다크 패턴이 아니라 투자(기록, 연속 기록, 저장 코스)에서 온다.
9. **걷는 이의 주의와 몸을 존중하라.** 앱은 산책을 돕지, 산책과 경쟁하지 않는다.
10. **기본이 신뢰할 수 있게.** 최소 데이터 수집, 정직한 알림, 안전을 고려한 추천.

## 2. 원칙 상세 & 디자인 함의

### P1 — 수 초 안에 걷기 시작
핵심 미충족 니즈는 *산책 전 결정 부담*이다. 디자인은 무엇보다 그 순간의 마찰을 없앤다.
- **하라:** 거의 입력 없는 기본 추천; 권한 직후 즉시 위치 사용; 폼 대신 프리셋 칩(10/20/30분); 단일 주 CTA("산책 시작").
- **하지 마라:** 가치를 보여주기 전에 가입, 긴 설문, 수동 주소 입력 요구. 게스트도 추천을 볼 수 있어야 한다.
- **목표:** 사용 가능한 추천까지 체감 시간 ≈ **3초**.
- **연결:** US-1.1, US-1.2, US-2.1.

### P2 — 한 번에 하나의 결정
선택 과부하가 적이다; 12개 코스 목록은 좋은 코스 하나보다 *더 나쁜* 경험이다.
- **하라:** **하나의** 주 추천 제시; "다른 코스 보기"를 단일하고 부담 없는 탈출구로 제공; 상세는 점진적으로 공개(지도/이유 먼저, 깊은 통계는 탭 시).
- **하지 마라:** 첫 화면에 순위 목록, 필터, 비교표를 전면에 배치.
- **연결:** US-2.1, US-2.4.

### P3 — 항상 이유를 설명하라
설명은 신뢰를 쌓고 "가장 가까운 길" 도구와 차별화한다 — 그리고 LLM 환각에 대한 가드레일이다.
- **하라:** **실제 속성**을 참조하는 한두 문장("강변, 조명 밝음, 대체로 평지, ~1.4km"); 사용자가 선택한 목적을 반영; 모든 주장을 코스 데이터에 근거.
- **하지 마라:** 데이터가 뒷받침 못 하는 분위기, 안전, 경관을 지어내기; 일반적 채움말("멋진 산책이 기다려요!") 사용.
- **디자인 규칙:** LLM은 *설명*을 쓰고; 지도 API는 *경로*를 소유한다. 카피는 코스 데이터에 대조해 검증 가능해야 한다.
- **연결:** US-2.2, PRD §6.2.

### P4 — 피드백은 탭이지 에세이가 아니다
피드백은 제품의 핵심 데이터 파이프라인이며, 마찰이 그것을 죽인다. **마찰 사다리:**

```
1. 원탭        👍 / 👎                             ← 여기서 ~80% 확보
2. 태그 선택   [🚧][🔇][🚫][🌑][⚠️]                ← 검증 데이터 대부분
3. 사진        📷                                  ← 소수 열성 사용자
4. 자유 텍스트 "…"                                  ← 드물지만 고품질
```
- **하라:** 1→2단계만으로 제품이 돌아가게; 3~4단계는 보너스로; 태그는 아이콘으로 한눈에; 모든 단계에서 건너뛰기 허용.
- **하지 마라:** 완료를 텍스트로 막기; 기본으로 키보드 열기; 👍/👎만 누른 사용자를 벌하기.
- **연결:** US-4.1, US-4.2, US-4.3.

### P5 — 적절한 순간에 물어라
하루 뒤의 *"산책 어땠나요?"*는 답을 못 받는다. 경험이 생생하고 맥락이 명확할 때 물어라.
- **하라:** **완주 즉시** 원탭 평가 표시; 문제 지점 바로 그 자리에서 **산책 중 핀 신고** 가능; 명확한 GPS 이탈 시 **자동 질문**("길 막힘/없음?").
- **하지 마라:** 피드백 요청을 몰아서 하기, 반복해서 조르기, 모달 다이얼로그로 걷는 중간을 끊기(프롬프트는 닫기 가능하고 비차단).
- **연결:** US-3.4, US-3.5, US-4.1.

### P6 — 기여가 기분 좋게 하라 (상호성 + 이타심)
두 동기 모두 존중:
- **상호성(이기적):** *"지난번 너무 조용하다고 하셔서 — 더 활기찬 경로로 골랐어요."* 내 피드백이 *내* 다음 산책을 눈에 띄게 개선.
- **이타심(임팩트):** *"당신의 '공사중' 신고가 12명이 이 길을 피하는 데 도움이 됐어요."*
- **하라:** 순환을 눈에 보이게 빠르게 닫아라; 임팩트를 정량화하라.
- **하지 마라:** 피드백을 블랙홀에 모으기; 게이밍을 부르는 포인트에 과의존.
- **연결:** US-2.5, US-7.4.

### P7 — 신뢰도에 정직하라
신뢰는 사용자가 알림을 켜두고 추천을 계속 믿는 이유다. MVP에서 *조용함*과 *관절친화* 데이터는 실제로 얇다.
- **하라:** 데이터가 희소한 곳에 신뢰도/한계 표시; 정직하게 유의점을 달기; MVP 지역에서는 강한 목적(안전, 자연)을 앞세우기.
- **하지 마라:** 데이터가 뒷받침 못 하는 목적을 과대 마케팅; 저신뢰 코스를 확실한 것처럼 제시.
- **디자인 규칙:** *"초기 마케팅에서 얇은 소음 데이터를 과대 포장하지 말라."* 정직함이 기능이다.
- **연결:** US-7.1, PRD §4, §7.

### P8 — 중독이 아니라 누적을 보상하라
리텐션은 **투자** 위에 세워진다 — 사용자가 더 많이 누적할수록(기록, 연속 기록, 저장 코스, 탐험 %) 떠날 때의 비용이 높아진다 — 조작적 메커니즘이 아니라.
- **하라:** 모든 산책을 자동 기록; 연속 기록과 월간 이정표를 축하; 코스 저장/명명/커스터마이즈 허용; 탐험/도감으로 발견을 유도(이는 피드백 수집도 겸함).
- **하지 마라:** 가짜 긴박감, 손실 미끼 알림, 어뷰징을 부르는 보상 스팸; 지역 밀도가 존재하기 전에 소셜 기능 출시(희소한 피드는 죽어 보인다).
- **연결:** US-6.1–6.4, US-6.3(도감 → 피드백), PRD §5.1F.

### P9 — 걷는 이의 주의와 몸을 존중하라
사용자는 *밖에서, 움직이며, 어쩌면 한 손으로, 햇빛 속에서, 어쩌면 차량에 정신이 팔린* 상태다. 앱은 그 현실에 맞아야 한다.
- **하라:** 끊임없는 턴바이턴보다 핵심 지점 안내; 큰 탭 타깃; 고대비, 햇빛 가독 비주얼; 한눈에 보는 상태; 비모달 산책 중 기록; **항상 이용 가능한 수동 "산책 종료."**
- **하지 마라:** 걸으며 정밀 타이핑 요구; 완벽한 GPS 가정; 사용자에게 화면을 계속 지켜보게 강요.
- **디자인 규칙:** 모든 산책 상태 결정에서 **GPS 오차**를 고려하라(허용 반경 + 정확도 값 + 수동 오버라이드) — 앱이 절대 "중단"을 잘못 기록하지 않도록.
- **연결:** US-3.1, US-3.2, US-3.3.

### P10 — 기본이 신뢰할 수 있게
- **하라:** 위치를 **최소한으로** 수집(산책에 필요한 것만; 불필요한 백그라운드 추적 없음); 메모에서 개인정보 필터링; 정말 걷기 좋은 날에만 푸시 발송(알림의 신뢰성 유지); 안전(조명, 횡단, 위험)을 추천과 주의 사항에 반영.
- **하지 마라:** 지속 추적, 무분별한 알림, 안전 데이터가 위험하다고 표시한 경로를 주의 없이 노출.
- **연결:** US-1.1, US-6.5, US-7.1, PRD §9.

## 3. 핵심 루프의 UX (단계별)

| 루프 단계 | UX 우선순위 | 핵심 동작 | 피해야 할 안티패턴 |
|---|---|---|---|
| **조건** | 속도 & 선택성 (P1) | 위치 우선; 프리셋 칩; 선택적 목적 태그; 합리적 기본값 | 폼, 강제 가입, 주소 타이핑 |
| **추천** | 명확한 선택 하나 + 이유 (P2, P3) | 주 코스 하나; 근거 있는 이유; 지도 + 거리 + 시간; "다른 거 보기" | 긴 목록, 전면 필터, 일반적 카피 |
| **산책** | 저주의, 몸 인지 (P9) | 핵심 지점 안내; 한눈에; 수동 종료; 비모달 메모 | 턴바이턴 과부하, 타이핑, GPS 과신 |
| **피드백** | 최저 마찰, 올바른 타이밍 (P4, P5) | 즉각 원탭; 태그 시트; 산책 중 핀; 이탈 시 자동 질문 | 텍스트 관문, 지연된 요청, 모달 잔소리 |
| **회고/개선** | 보상 + 정직 (P6, P7) | 따뜻하고 진실한 회고; 눈에 보이는 "당신 피드백이 X를 했어요"; 신뢰도 표시 | 지어낸 사건, 블랙홀 피드백, 과잉 약속 |
| **재방문** | 중독 아닌 투자 (P8, P10) | 연속 기록, 도감, 저장 코스, 좋은 날 푸시 | 다크 패턴, 스팸 푸시, 조급한 소셜 |

## 4. 콘텐츠 & 보이스 가이드라인
- **톤:** 따뜻하고, 격려하며, 간결하게 — 코치나 마케터가 아니라 친근한 동네 사람.
- **근거 있게, 과장 없이:** 실제인 것을 묘사("강변, 그늘짐, 평지"), 데이터가 뒷받침 못 하는 최상급은 피하라.
- **2인칭, 현재형, 실행 가능하게:** "여기서 20분 순환, 준비되셨나요?"가 "산책 경험이 기다립니다"보다 낫다.
- **한계에 정직하게:** "이곳이 얼마나 조용한지는 우리도 덜 확실해요"가 거짓 확신보다 낫다.
- **가능하면 임팩트를 숫자로:** "12명이 이 길을 피하는 데 도움" > "피드백 감사합니다."
- **회고는 부드러운 요약처럼 읽히게**, 결코 지어낸 일기가 아니게.

## 5. 접근성 & 환경 적합성
- **야외 가독성:** 고대비, 큰 타이포, 햇빛에서 검증된 색상 선택; 색상에만 의존하지 말 것(아이콘/라벨과 짝지어라 — 목적 태그와 신뢰도 상태에서 중요).
- **한 손, 이동 중:** 엄지가 닿는 주 동작; 넉넉한 탭 타깃; 최소한의 정밀 제스처.
- **중단 가능:** 어떤 기록(메모, 피드백)도 일시정지/재개 가능; 무엇도 산책을 막지 않는다.
- **환경에 관대하게:** 나쁜 GPS, 거부된 권한, 경로 중간의 오프라인 구간에서도 우아하게.
- **포용적 대상:** 로드맵은 사회적 취약 계층(예: 어르신, 재활)을 포함한다 — 단순함, 더 큰 터치/텍스트, 낮은 인지 부하를 사후 고려가 아니라 기본값으로 선호하라.

## 6. UX 측정 (디자인이 효과가 있는가?)
정성적 원칙을 PRD의 퍼널에 연결해 디자인 결정을 검증 가능하게 하라:
- **P1/P2 →** 산책 시작률, 코스 선택률, 추천까지의 시간.
- **P3/P6 →** 재사용 의향, "다른 코스 보기" 비율(낮을수록 = 더 잘 맞음), 회고 만족도.
- **P4/P5 →** 메모/피드백률, 마찰 사다리 분포(1~2단계에서 확보하고 있는가?), *완주 시점*에 주어진 피드백 비율 vs. 이후.
- **P7 →** 신뢰 프록시: 알림 해제율, 얇은 데이터 목적에 대한 추천 수용률.
- **P8/P9/P10 →** 완주율, 재방문율, 연속 기록 유지, 수동 종료 사용(급증은 GPS/UX 문제 신호일 수 있음), 알림 옵트인 유지.

## 7. 디자인 가드레일 (강제 규칙)
1. 절대 계정 생성 뒤로 "산책 시작"을 막지 말 것.
2. 절대 근거 있는 이유 없이 추천을 보여주지 말 것.
3. 절대 피드백 완료에 타이핑을 요구하지 말 것.
4. 절대 데이터가 뒷받침 못 하는 코스 속성을 주장하지 말 것.
5. 절대 정말 걷기 좋은 날/시간이 아니면 푸시를 보내지 말 것.
6. 항상 수동 "산책 종료"를 제공할 것.
7. 항상 중단/완주를 기록하기 전에 GPS 오차를 고려할 것.
