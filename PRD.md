## 1. 개요

### 1.1 한 줄 소개
산책온은 **위치 기반 AI 산책 코스 추천 서비스**입니다. 사용자의 *현재 위치*와 *산책 조건*(가용 시간, 거리, 난이도, 분위기, 안전)에 맞는 산책 경로를 추천하고, 산책 중 남긴 메모와 피드백을 AI가 생성한 회고와 코스 품질 개선 데이터로 전환합니다.

### 1.2 문제 정의
걷기는 일상에서 가장 시작하기 쉬운 활동 중 하나이지만, 막상 시작하기 전에 사용자는 **어디를** 걸을지 스스로 정해야 합니다. 좋은 선택은 여러 요인 — 가용 시간, 목적, 선호하는 분위기, 예상 거리, 보행 환경 — 에 동시에 좌우되므로, 단순한 최단 경로나 "가장 가까운 길" 답변만으로는 충분하지 않습니다.

기존 제품들은 이 "산책 전 코스 선택" 순간을 해결하지 못합니다:
- **지도 / 내비게이션 앱**은 목적지 경로 안내에는 강하지만, *"지금 걷기 좋은 순환 코스 하나 추천해줘"* 에는 약합니다.
- **걷기 / 러닝 / 트레킹 앱**은 기록, 성과, 보상 중심입니다 — 기록은 이미 어디로 갈지 정한 *이후*에 남습니다.
- **보상형 앱**(포인트, 캐릭터 육성, 광고)은 사용자의 현재 상황에 맞는 코스가 아니라 참여 유도 메커니즘을 최적화합니다.
- **피드백이 순환을 완성하는 경우가 드뭅니다:** 유사 서비스 대부분은 리뷰나 로그를 수집하지만, 그 피드백이 더 나은 추천으로 되돌아 흐르는 일은 거의 없습니다.

### 1.3 해결책
산책온은 산책 *시작*의 결정 부담을 없애고 *끝*에서 순환을 완성합니다:
1. **추천** — 현재 위치와 선택한 조건으로부터 AI가 요청을 해석하고, 공공 데이터 + 지도 API로 후보 코스를 조합하며, 지도 API로 거리/시간/경유지를 검증한 뒤 **자연어 추천 이유와 함께** 적합한 코스를 추천합니다.
2. **산책** — 사용자는 코스의 핵심 지점(출발, 경유지, 반환점, 도착)을 따라 걸으며 도중에 간단한 메모나 탭 기반 피드백을 남깁니다.
3. **회고 & 개선** — 완주 시 AI가 산책 기록 + 메모 + 피드백으로부터 개인 회고를 생성하고, 향후 추천을 개선하는 코스 품질 태그(분위기, 길 상태, 위험 요소, 주의 사항)를 추출합니다.

### 1.4 전략적 명제 (왜 이길 수 있는가)
> **이것은 데이터 검증 제품이다. 리텐션 루프가 곧 피드백 루프다.**

공공 데이터는 수만 개의 노드를 배치할 수 있고 지도 API는 걸을 수 있는 경로를 생성할 수 있지만, **데이터만으로는 그 길이 실제로 쾌적한지, 아니 걸을 수는 있는지조차 검증할 수 없습니다**(공사, 막다른 길, 소음, 어둠, 위험). 따라서 **사용자 피드백이 제품의 핵심 데이터 파이프라인입니다.** 제품 전반의 설계 원칙은 다음과 같습니다:

```
[걷고 싶은 마음] → [경로 추천] → [실제 산책] → [경로 피드백] → [데이터 개선]
       ↑                                                                     │
       └──────────── 더 나은 추천이 재방문을 이끈다 ←─────────────────────────┘
```

*피드백을 남기는 행위 자체가 사용자에게 이롭게* 만들면, 리텐션과 데이터 수집이 함께 해결됩니다.

## 2. 목표 & 비목표

### 2.1 제품 목표
| # | 목표 | 근거 |
|---|------|-----------|
| G1 | 사용자가 "걷고 싶다"에서 "선택한 코스를 걷는다"까지 결정 마비 없이 **수 초 안에** 도달하게 한다. | 핵심 미충족 니즈. 약 3초 안에 강력한 추천 하나를 보여준다. |
| G2 | 추천을 **설명 가능하게** 만든다(실제 코스 데이터에 근거한 자연어 이유). | 신뢰 + 차별화; "블랙박스"와 LLM 환각을 방지. |
| G3 | 모든 산책을 최소한의 사용자 마찰로 **구조화된 코스 품질 데이터**로 전환한다. | 추천을 시간이 갈수록 개선하는 데이터 파이프라인. |
| G4 | 확장 전에 한 지역(대전)에서 **검증된 코어**를 구축한다. | 밀도를 통해 콜드 스타트 문제를 해결. |
| G5 | 무거운 참여 메커니즘 없이도 사용자가 돌아오는 **습관 루프**를 만든다. | 투자(누적된 기록/피드백/저장 코스)를 통한 리텐션. |

### 2.2 성공 지표 (제안서의 검증 계획 기준)
주요 퍼널 (제한된 테스트 지역에서 먼저 측정):
- **산책 시작률** — 사용자가 실제로 추천 산책을 시작한 세션.
- **코스 선택률** — 추천이 선택된 비율 vs. 넘긴 비율("다른 코스 보기").
- **완주율** — 끝까지 마친 산책 vs. 중단한 산책.
- **메모/피드백률** — 최소한 원탭 평가나 메모를 받은 산책.
- **회고 만족도** — AI 회고에 대한 사용자 평가.
- **재사용 의향 / 재방문율** — 다시 걸을 의향; 반복 세션.

데이터 파이프라인 건강도 (제품 특화):
- 최소 1개 이상의 피드백 신호(명시적 또는 암묵적)를 가진 엣지/구간의 비율.
- 신뢰도 커버리지: 추천 코스 길이 중 "검증됨"(다중 사용자) 신호로 뒷받침되는 %.
- 피드백 마찰: 마찰 사다리(탭 → 태그 → 사진 → 텍스트)에 걸친 분포.

### 2.3 비목표 (MVP 범위에서 명시적으로 제외)
- **전국 커버리지.** MVP는 대전 한정; 콜드 스타트에서는 넓이보다 밀도가 우선.
- **라우팅 엔진으로서의 턴바이턴 내비게이션.** 지도 API가 경로를 계산하고, 우리는 핵심 지점을 제시하고 진행을 모니터링할 뿐 내비 앱을 대체하지 않는다.
- **무거운 소셜 네트워크 / 피드.** 커뮤니티는 3단계이며, 지역 밀도가 존재한 후에만(희소한 소셜 피드는 텅 비어 보인다).
- **출시 시점의 보상 경제(포인트 → 현금/쿠폰).** 어뷰징 방지 장치와 함께 추후 검토; 핵심 루프에는 불필요.
- **라우터로서의 LLM.** LLM은 기하 계산을 **하지 않는다.** 후보 중에서 선택하고 설명/회고/태그만 생성한다(멘토 지침, 환각 방지).

## 3. 대상 사용자 & 맥락

### 3.1 주요 사용자
**걷고 싶지만 현재 위치에서 좋은 코스를 고르기 어려워하는** 사람들 — 기분 전환, 스트레스 해소, 가벼운 운동, 건강을 위해 걷지만, 그렇지 않으면 같은 길만 반복하거나 목적 없이 헤매는 이들.

### 3.2 대표 상황
- 점심 후 / 퇴근 후: *"20분 있는데, 여기서 걷기 좋은 순환 코스 어디 없나?"*
- 날씨 좋고 공기 좋은 아침: *"오늘은 걷기 좋은 날이네."*
- 변화를 원할 때: *"같은 거리는 이제 지겨워."*

### 3.3 보조 사용자 / 확장
- 도시의 생활권 산책로와 걷기 프로그램을 이용하는 **지역 주민 & 방문객.**
- **지자체 / 기관**(추후): 누적된 산책 데이터로 구축한 걷기 프로그램 운영 도구와 보행 환경 피드백 리포트.

## 4. 산책 목적 (추천 차원)

산책온은 추천을 **네 가지 산책 목적**을 중심으로 구성하며, 각 목적은 구체적인 데이터 레이어로 뒷받침됩니다. MVP 데이터 강도(대전)는 데이터풀 보고서 기준으로 목적별로 표기했습니다.

| 목적 | 의미 | 뒷받침 데이터 (대전 MVP) | 데이터 강도 |
|---|---|---|:---:|
| 🌳 **자연친화** | 녹지, 경관, 숲길 느낌 | 대전 둘레산길(국가숲길, 12개 구간 ≈138km, GPX); 도시 자전거/보행로 | **강점(고유 차별화 요소 — 후보 3개 도시 중 대전에만 국가숲길이 있음)** |
| 🛡️ **안전** | 밝고, 감시되며, 횡단 위험이 낮음 | 가로등(위경도 포함 43,082개), CCTV(시+구), 보행로 표준 데이터(폭, CCTV, 보안등, 보차 분리), 횡단보도 + 보행자 사고 통계 | **강점(가장 두터운 풀)** |
| 🦵 **관절친화** | 저충격, 계단/급경사 회피 | TMAP `turnType` 태그: 육교(125), 지하보도(126), 경사로(128), 엘리베이터(218); 트레일 경사 | 부분적 — **계단이 공백**(TMAP는 계단을 반환하지 않음; OSM `highway=steps`/`incline` 필요) |
| 🤫 **조용함** | 낮은 소음, 차분한 거리 | 국가 소음 도로점 데이터(대전: 단 6개 지점, 주소만, 좌표 없음) | **가장 얇은 풀 — 최대 공백**(지오코딩 + 격자 소음 데이터 필요; 출시 시 과대 마케팅 금지) |

**MVP 포지셔닝에 대한 시사점:** **안전 + 자연친화**를 앞세우고(데이터가 실제로 두텁다), **관절친화**와 **조용함**은 정직한 신뢰도 표시와 함께 최선의 노력으로 다루며, 시간이 지나면서 사용자 피드백으로 공백을 메운다.

## 5. 기능 요구사항

### 5.1 MVP 기능 세트 (필수)
핵심 루프별로 그룹화. 우선순위: **P0** = MVP 필수, **P1** = MVP 기간 내 강력히 희망, **P2** = 이후 단계.

#### A. 계정, 위치 & 조건
- **A1 (P0)** 권한 처리 및 정확도 값과 함께 현재 위치(GPS)를 획득.
- **A2 (P0)** 최소한의 계정/인증(기록 누적에 필요 — 리텐션을 이끄는 투자). 위치는 최소한으로 수집.
- **A3 (P0)** 조건 입력: 가용 시간, 원하는 거리, 난이도, 그리고 **목적 태그**(자연 / 조용함 / 관절친화 / 안전). 사용자가 거의 입력 없이도 코스를 받을 수 있도록 합리적인 기본값 제공.

#### B. 추천
- **B1 (P0)** AI가 후보 코스를 생성: 공공 데이터 + 지도 API에서 인근 산책로 / 공원 / 보행로 / 핵심 지점을 조회하고, 후보를 조합하며, **지도 API로 거리, 예상 시간, 경유지를 검증.**
- **B2 (P0)** 선택 마비를 막기 위해 **강력한 대표 추천 하나를 빠르게**(약 3초 목표) 반환; "다른 코스 보기" 허용.
- **B3 (P0)** 실제 코스 데이터에 근거한 **자연어 추천 이유**(꾸며낸 속성 금지).
- **B4 (P0)** 코스 상세: 핵심 지점(출발, 경유지, 반환점, 도착), 거리, 예상 시간, 목적 적합도, 그리고 데이터가 얇은 곳의 **신뢰도/한계 표시.**
- **B5 (P1)** 추천 정교화: *"지난번 '너무 조용했다'는 피드백을 반영해서 이번엔 다르게 골랐어요"* — 피드백의 효과를 눈에 보이게.

#### C. 산책 진행
- **C1 (P0)** 핵심 지점 기반 산책 진행: 출발 → 경유지 → 반환점 → 도착을 따라 안내하며 위치 추적.
- **C2 (P0)** GPS를 통한 완주 / 이탈 감지 — **핵심 지점 허용 반경**, **위치 정확도 값** 활용, 그리고 (GPS 오차 대응을 위한 — 프로젝트에서 식별된 리스크) **수동 종료** 옵션 포함.
- **C3 (P0)** 산책 중 메모 기록: 자유 한 줄 메모와 선택형 피드백.
- **C4 (P1)** **산책 중 핀 드롭**("여기 문제 있어요") → 현재 GPS 위치에 핀을 찍고 가장 가까운 엣지/구간에 매핑(최고 품질의 구간 단위 데이터).
- **C5 (P1)** **"여긴 길이 없어요" 자동 감지:** 사용자의 GPS가 경로에서 벗어나거나 되돌아가면 *"길이 막혔거나 존재하지 않나요?"* 라고 질문 — 지도/공공 데이터 오류를 잡는 가장 가치 있는 신호.

#### D. 피드백 (데이터 파이프라인)
- **D1 (P0)** 완주 즉시 **원탭 산책 후 평가**: 기억이 생생할 때 👍 / 👎.
- **D2 (P0)** 👎 시 (또는 항상 이용 가능한) **태그 시트**: `🚧 공사중`, `🔇 시끄러움`, `🚫 길없음`, `🌑 어두움`, `⚠️ 위험`, `+ 직접입력`. 마찰 사다리의 1→2단계 태그가 검증 데이터 대부분을 담당.
- **D3 (P1)** 사진 첨부(📷)와 자유 텍스트 — 보너스, 고품질, 열성 사용자로부터.
- **D4 (P0, 백엔드)** **암묵적 피드백 수집**(사용자 노력 제로): 완주 vs. 중단, GPS 이탈 + 우회, 잦은 멈춤 / 되돌아감, 즉각적인 "다른 코스 보기", 저장/재산책 비율. 명시적 피드백이 부족한 콜드 스타트 시기에 결정적.

#### E. AI 분석 & 코스 품질
- **E1 (P0)** 완주 시 산책 기록 + 메모 + 피드백으로부터 생성되는 **AI 회고.**
- **E2 (P0)** 메모/피드백에서 **사전 정의된 태그 분류 체계**로의 **태그 추출**(분위기, 길 상태, 문제 구간, 주의 사항) — 분류 체계는 사전에 정의하며(멘토 지침), 태그 → 다음 추천으로 이어지는 정량화된 경로 포함.
- **E3 (P0)** 코스 주의 사항과 추천 우선순위에 반영되는 **위험/불편 요소 추출.**
- **E4 (P0)** **신뢰도 & 교차 검증:** 단일 "공사중" 신고 = *의심*; 3건 이상 = *확정*. 신뢰도 점수를 적용하고, 해당 구간을 GPS로 실제 완주한 사용자의 신고에 가중치를 주며, 명시적 + 암묵적 신호를 결합.
- **E5 (P1)** **시간 감쇠 & 재검증:** 공사 신고는 약 2~3개월 후 자동 만료 → *"아직도 공사 중인가요?"*
- **E6 (P1)** **어뷰징 필터링:** 걷지 않은 구간을 반복적으로 신고하는 계정을 탐지.

#### F. 리텐션 기능 (전략 문서 기준, 우선순위화)
- **F1 (P0)** **산책 기록 & 연속 기록**(투자): 경로, 거리, 시간, 예상 칼로리 자동 기록; 연속 기록("이번 주 3회 산책 / 5주 연속")과 월간 리포트 표시. 저비용, 고효과 — 리텐션의 근간.
- **F2 (P1)** **"대전 탐험" 컬렉션 / 도감:** 탐험 %가 있는 지역 정복 지도, 희귀도 등급이 있는 경로 도감, 배지. 이 메커니즘은 또한 **사용자를 아직 검증되지 않은 경로로 유도 → 자연스럽게 피드백을 모은다.**
- **F3 (P1)** **코스 저장 & 커스터마이즈:** 마음에 드는 경로를 저장, 이름 붙이기, 가볍게 편집 — 투자를 누적.
- **F4 (P1)** **맥락 트리거:** 날씨/대기질이 좋은 날 아침 푸시(기존 공공 데이터 파이프라인 재사용; 정말 걷기 좋은 날에만 발송); 시간대별 제안("10/20/30분 코스").
- **F5 (P2)** **기여 임팩트 & 등급:** *"당신의 '공사중' 신고가 12명이 이 길을 피하는 데 도움이 됐어요,"* 기여자 등급(새싹 → 탐험가 → 대전 길잡이), 지역 리더보드 — 어뷰징 방지(E4–E6)와 병행.
- **F6 (P2)** **커뮤니티 / 지역 리뷰, "이번 주 34명이 이 길을 걸었어요,"** 지자체 제휴 보상 — 지역 밀도가 존재한 후에만.

## 6. 시스템 & 아키텍처

### 6.1 산출물 (제안서 기준)
| 산출물 | 핵심 기능 |
|---|---|
| **사용자 모바일 웹/앱** | 코스 추천, 산책 진행, 자유 메모, AI 회고 |
| **어드민 웹 대시보드** | 코스 관리, AI 분석 검토, 통계 |
| **Spring API 서버** | 인증, 코스 추천, 산책 기록, 메모/피드백 저장 |
| **AI 분석 모듈** | 이유 생성, 회고 생성, 태그 & 위험 추출 |

### 6.2 컴포넌트 책임
- **지도 API (TMAP 보행자):** 실제 경로 기하, 거리, 시간, 시설/회전 태그를 계산. **LLM이 아니라 이것이 라우팅을 담당한다.**
- **공공 데이터 레이어:** 코스 후보와 스코어링 입력의 시드(숲길, 가로등, CCTV, 보행로, 소음).
- **LLM (분석 모듈):** 지도로 검증된 옵션 중 후보 **선택**, 자연어 **이유**, **회고** 생성, **태그/위험 추출** — 환각 방지를 위해 실제 코스 데이터에 엄격히 근거.
- **피드백 + 암묵 신호 저장소:** 신고/신호를 엣지/구간에 매핑; 재랭킹을 위한 구간별 신뢰도 유지.

### 6.3 개발 환경 (제안서 기준)
- **OS:** Ubuntu 22.04 LTS · **도구:** IntelliJ, VS Code, GitHub, Docker
- **언어:** Java 21 (Spring Boot 백엔드), TypeScript (React 프론트엔드), Python (AI 모듈)
- **AI 모델:** 메모 분석 및 회고 생성을 위한 LLM API
- **데이터:** 초기 후보 풀은 공공 데이터 + 지도 API; 피드백 기반 품질 개선; 임베딩/유사도용 `pgvector`(멘토 승인).

### 6.4 아키텍처 노트 / 멘토 지침
- **역할 분리(환각 방지):** 지도 API가 경로를 계산하고; LLM이 선택 & 설명하며; 이유는 실제 코스 데이터에 근거해야 한다.
- **안정성:** 단일 EC2는 데모 안정성을 위협하는 SPOF다. 비동기 AI 회고/태그 생성을 요청 경로에서 분리(예: EC2 내 `@Async` 대신 **SQS + Lambda**)하고, 최소한 ALB + 멀티 AZ를 검토하라. "간단/저비용"이 목표라면 서버리스(API Gateway + Lambda)를 평가하라.
- **사전 정의된 태그 분류 체계**와, 각 태그가 다음 추천을 어떻게 바꾸는지에 대한 **정량화된** 매핑.

## 7. 데이터 현황 & 의존성 (MVP = 대전)

실제 API 호출 / 다운로드로 검증됨(데이터 보고서 참조). 준비된 것 vs. 작업이 필요한 것 요약:

| 목적 | 즉시 사용 가능 | 필요 조건 / 조치 | 판정 |
|---|:---:|---|:---:|
| 안전 (조명/CCTV) | ✅ 가로등 4.3만 + 다층 CCTV | 그대로 사용 | **강점** |
| 보행/산책로 | ✅ 표준 데이터 + TMAP | 보행로 표준 데이터셋 이용 신청 1건(전용 serviceKey 발급) | **강점** |
| 자연친화 | ✅ 둘레산길 12개 구간 | GPX 지금 사용; POI는 제공처 502 장애 복구 후 | **강점(차별화 요소)** |
| 관절친화 | △ `turnType` 태그 | 계단 보조 데이터 필요(OSM `steps`/`incline`) | **공백 보완** |
| 조용함 | △ 소음 6개 지점 | 지오코딩 + 격자 소음 데이터 | **최대 공백** |

**알려진 데이터 유의점(왜곡 표현 금지):**
- **TMAP는 계단을 반환하지 않는다**(`turnType 127` / `facilityType 16`) — 알려진 계단 위치에서조차 → 계단은 보조 데이터가 필요. 신뢰할 수 있는 물리 시설 신호는 `facilityType`이 아니라 **`turnType`**이다(문서화된 코드표는 부정확했음).
- **국가 소음 데이터는 dB는 있으나 좌표가 없다**(주소 텍스트만) → 지오코딩하거나 격자 기반 소음 데이터로 전환.
- **국가숲길 POI API는 현재 제공처 백엔드에서 502** → 복구 전까지 GPX + 지정 목록 사용.
- **보행로 표준 데이터**는 필드가 완벽하지만 **현재 키에 등록되어 있지 않다** → 별도 이용 신청 필요.

### 7.1 즉시 조치 항목
- [ ] 전국 보행로 표준 데이터 이용 신청 제출(전용 serviceKey 획득).
- [ ] 국가숲길 POI API 502 복구 모니터링.
- [ ] 소음 데이터를 격자 형태로 교체하거나 주소 지오코딩 파이프라인 구축.
- [ ] 계단/경사 보조 데이터 확보(OSM `steps`/`incline`, 도로대장 종단 경사).
- [ ] 코스 로직에서 라우팅/LLM 역할 분리 강제.

## 8. 출시 계획 (콜드 스타트를 고려한 로드맵)

### 1단계 — 데이터 검증 MVP (지금)
- 대전의 **한 밀집 지역**(예: 유성구 또는 갑천 축)에 집중.
- 출시: **원탭 산책 후 피드백 + 산책 중 핀 신고 + 암묵적 GPS 신호 수집.**
- 리텐션: 우선 **산책 기록만**(연속 기록/기록 = 저비용, 고효과).
- 목표: **검증된 코어** — *"적어도 이 지역에서는 경로를 신뢰할 수 있다."*

### 2단계 — 리텐션 강화
- 날씨/대기질 트리거 푸시, 탐험 도감 / 지역 정복, 계절 큐레이션.
- 기여자 등급 + 어뷰징 방지 장치.

### 3단계 — 확장 & 커뮤니티
- 검증된 지역을 대전 전역으로, 이후 다른 도시로 확장(확장 순서: **서울** — 가장 풍부한 보행 인프라 & 소음 지도, 2순위 → **부산** — 급경사/계단 데모에는 좋으나 데이터 공백).
- 지역 리뷰 / 소셜, 지자체 제휴 보상.

### 8.1 제안서 타임라인 (참고)
기획/분석/설계 → MVP 개발(인증+위치, 추천, 코스 상세, 핵심 지점 산책, 자유 메모) → AI 회고 + 메모 분석 + 코스 자동 태그/주의 사항 + 어드민 웹 → 어드민 대시보드/통계 → 제한 지역 현장 테스트 → 1차 MVP 배포(제한 지역) → 최종 배포 + 데모 자료.

## 10. 비즈니스 모델 & 임팩트

- **사용자:** 현재 위치에서 지금 걸을 코스를 얻고, *왜* 추천됐는지 이해하며, AI 회고를 통해 산책 경험을 기록한다.
- **서비스:** 사용자 메모가 코스의 분위기 / 안전 / 길 상태 정보를 풍부하게 하여 추천 품질을 개선한다 — 복리로 쌓이는 데이터 해자.
- **수익화:** 우선 무료 서비스로 사용성을 검증하며, 배너 광고와 산책 후 짧은 광고; 이후 누적된 산책 + 피드백 데이터가 **지자체 걷기 프로그램 운영 도구와 보행 환경 피드백 리포트**가 된다(기관 제휴 수익). 사회 공헌 측면도 존재한다(저소득 어르신 / 재활 대상 걷기 프로그램).

## 11. 미해결 질문
- 대전 내 밀도 확보를 시작할 정확한 세부 지역(유성구 vs. 갑천 축).
- 정확한 재랭킹 공식: 확정된 각 태그가 구간 점수를 얼마나 이동시키는가.
- 데모 안정성 요구를 고려한 비동기 AI 처리 대상(EC2 내 `@Async` vs. SQS+Lambda).
- 마찰과 투자 누적 사이에서 균형을 맞추는 최소 실행 가능 인증(게스트 모드 vs. 필수 가입).
- 신뢰도 표시를 명시적 UI 요소로 노출할지, 추천 이유에 녹여 넣을지.
